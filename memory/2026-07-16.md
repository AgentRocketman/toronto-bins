### ffmpeg Crossfade Offset Bug — **WHY VIDEOS ONLY SHOW 2 IMAGES**

**THE REAL REASON** every 25-clip video only showed ~2 clips of content:

`stitch_group()` function computed crossfade offsets as:
```python
offset = i * (clip_dur - fade_s)  # i * (5.0 - 0.8) = i * 4.2
```

This assumes every input clip is exactly 5 seconds. But when used recursively to merge **batches**, each batch file is ~26 seconds (6 clips × 5s - 5 fades). The offset formula crossfaded to batch 2 at 4.2s instead of 25.2s — clipping off ~22 seconds of each batch.

**Evidence**: `l0_b0_stitched.mp4` through `l0_b3_stitched.mp4` all ~26s (correct). `l0_r_stitched.mp4` = 21.83s (should be ~106s). `final_output.mp4` = 6.87s. This matches Christ's report of "only 2 images visible."

### Fix Applied to generate_ai_walkthrough.py

`stitch_group()` now probes each input's actual duration via `ffprobe` before building filtergraph:
```python
durations = []
for c in clips:
    probe = subprocess.run(['ffprobe', '-v', 'quiet', '-show_entries', 
        'format=duration', '-of', 'default=noprint_wrappers=1:nokey=1', c],
        capture_output=True, text=True)
    durations.append(float(probe.stdout.strip() or clip_dur))

offsets = [0.0]
for i in range(1, g):
    offsets.append(offsets[i-1] + durations[i-1] - fade_s)
```

## 🎬 ffmpeg Subtitle Burn-In — libass Missing, Python Overlay Workaround (2026-07-16 ~13:14 UTC)

### The Problem
Every subtitle render in `generate_ai_walkthrough.py` failed with `AVFilterGraph … No option name near …` / `Error parsing filterchain`. The `subtitles` filter didn't exist at all in the ffmpeg binary.

**Root cause**: The Homebrew bottled ffmpeg (`v8.1.2_1`) was compiled **without `--enable-libass`**. The `subtitles` filter literally doesn't exist. This is why Ken Burns videos always worked — `generate_kenburns.py` never uses the `subtitles` filter.

### What was tried & failed
1. `brew install ffmpeg-full` — partially completed but produced a broken binary with `@@HOMEBREW_PREFIX@@` linker paths. Multiple retries hit process locks and were killed (Docker's weak CPU makes source builds impractical).
2. Multiple ffmpeg filter syntax variations (quotes, no quotes, `filename=`) — all failed because the filter itself doesn't exist.
3. `-pix_fmt yuva420p` for alpha-channel subtitle video → silently rejected by libx264 encoder, resulting in opaque black subtitles.

### The Fix: Python Pillow Overlay Pipeline
Wrote `burn_subs_overlay.py` that works WITHOUT libass:

1. **Parse** ASS file → extract (start_time, end_time, text) tuples
2. **Render** each subtitle as a full-frame RGBA PNG with Pillow (white text, drop shadow, transparent bg)
3. **Concat** PNGs into a single subtitle video using ffmpeg concat demuxer
4. **Overlay** using `colorkey` to make black transparent → final video with burned subtitles

Key: only 2 inputs to ffmpeg (main video + subtitle video), avoiding pthread exhaustion.

### Manual Test Verified
- Job `6c25c4e80b703840fe62f334abae6aa9`: 22s, 854x480, subtitles clearly visible over AI walkthrough video
- Chris confirmed: subtitles visible on the video

### Files created
- `/agentado/api/video/burn_subs_overlay.py` — complete subtitle burn-in module

### Integration
`generate_ai_walkthrough.py` is modified to:
1. Merge video + audio (no subtitles) → `merged_no_subs.mp4`
2. If ASS file exists, call `burn_subs_overlay.py merged_no_subs.mp4 subs.ass final_output.mp4`
3. Else just rename `merged_no_subs.mp4` → `final_output.mp4`

### Cleanup Strategy

### ffmpeg Crossfade Offset Bug — **WHY VIDEOS ONLY SHOW 2 IMAGES**

**THE REAL REASON** every 25-clip video only showed ~2 clips of content:

`stitch_group()` function computed crossfade offsets as:
```python
offset = i * (clip_dur - fade_s)  # i * (5.0 - 0.8) = i * 4.2
```

This assumes every input clip is exactly 5 seconds. But when used recursively to merge **batches**, each batch file is ~26 seconds (6 clips × 5s - 5 fades). The offset formula crossfaded to batch 2 at 4.2s instead of 25.2s — clipping off ~22 seconds of each batch.

**Evidence**: `l0_b0_stitched.mp4` through `l0_b3_stitched.mp4` all ~26s (correct). `l0_r_stitched.mp4` = 21.83s (should be ~106s). `final_output.mp4` = 6.87s. This matches Christ's report of "only 2 images visible."

### Fix Applied to generate_ai_walkthrough.py

`stitch_group()` now probes each input's actual duration via `ffprobe` before building filtergraph:
```python
durations = []
for c in clips:
    probe = subprocess.run(['ffprobe', '-v', 'quiet', '-show_entries', 
        'format=duration', '-of', 'default=noprint_wrappers=1:nokey=1', c],
        capture_output=True, text=True)
    durations.append(float(probe.stdout.strip() or clip_dur))

offsets = [0.0]
for i in range(1, g):
    offsets.append(offsets[i-1] + durations[i-1] - fade_s)

## 🎬 ffmpeg Subtitle Burn-In — libass Missing, Python Overlay Workaround (2026-07-16 ~13:14 UTC)

### The Problem
Every subtitle render in `generate_ai_walkthrough.py` failed with `AVFilterGraph … No option name near …` / `Error parsing filterchain`. The `subtitles` filter didn't exist at all in the ffmpeg binary.

**Root cause**: The Homebrew bottled ffmpeg (`v8.1.2_1`) was compiled **without `--enable-libass`**. The `subtitles` filter literally doesn't exist. This is why Ken Burns videos always worked — `generate_kenburns.py` never uses the `subtitles` filter.

### What was tried & failed
1. `brew install ffmpeg-full` — partially completed but produced a broken binary with `@@HOMEBREW_PREFIX@@` linker paths. Multiple retries hit process locks and were killed (Docker's weak CPU makes source builds impractical).
2. Multiple ffmpeg filter syntax variations (quotes, no quotes, `filename=`) — all failed because the filter itself doesn't exist.
3. `-pix_fmt yuva420p` for alpha-channel subtitle video → silently rejected by libx264 encoder, resulting in opaque black subtitles.

### The Fix: Python Pillow Overlay Pipeline
Wrote `burn_subs_overlay.py` that works WITHOUT libass:

1. **Parse** ASS file → extract (start_time, end_time, text) tuples
2. **Render** each subtitle as a full-frame RGBA PNG with Pillow (white text, drop shadow, transparent bg)
3. **Concat** PNGs into a single subtitle video using ffmpeg concat demuxer
4. **Overlay** using `colorkey` to make black transparent → final video with burned subtitles

Key: only 2 inputs to ffmpeg (main video + subtitle video), avoiding pthread exhaustion.

### Manual Test Verified
- Job `6c25c4e80b703840fe62f334abae6aa9`: 22s, 854x480, subtitles clearly visible over AI walkthrough video
- Chris confirmed: subtitles visible on the video

### Files created
- `/agentado/api/video/burn_subs_overlay.py` — complete subtitle burn-in module

### Integration
`generate_ai_walkthrough.py` is modified to:
1. Merge video + audio (no subtitles) → `merged_no_subs.mp4`
2. If ASS file exists, call `burn_subs_overlay.py merged_no_subs.mp4 subs.ass final_output.mp4`
3. Else just rename `merged_no_subs.mp4` → `final_output.mp4`

### Cleanup Strategy
Temporary subtitle PNGs and concat files are created in `tempfile.gettempdir()` and cleaned up after processing.
