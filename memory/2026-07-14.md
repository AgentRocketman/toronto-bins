## 🎬 Free Cinematic Walkthrough Option — DepthFlow

Chris asked for a free way to achieve cinematic walkthroughs in Agentado.

### Recommendation: DepthFlow
- **What**: Open-source Python + GLSL shader video generator
- **What it does**: Takes a photo → depth map estimation → 3D parallax fly-through video with true depth perception (foreground moves faster than background)
- **Output**: Up to 8K, smooth camera movement
- **License**: Commercial use allowed
- **Stars**: ~1.5k on GitHub, active project
- **Repo**: `patricktrainer/depthflow` on GitHub

### Difference from Ken Burns
Ken Burns = flat 2D zoom/pan on a photo
DepthFlow = 3D camera moving through a scene with parallax depth — foreground objects shift faster than background, creating immersive cinematic feel. Much closer to Trolto's look.

### Free Hosting Options
1. **Google Colab** (free T4 GPU) — runs DepthFlow natively, expose via Gradio/Flask API. Pro: zero cost. Con: sessions timeout after ~2-3h idle, ~12h max.
2. **Hugging Face Space** (free CPU tier) — always-on but slower without GPU.
3. **RunPod spot instance** (~$0.20/hr) — cheap fallback if Colab unreliable.

### Integration Plan
```
Current pipeline: Photos → Ken Burns pan/zoom → Agentado overlays → Video
Proposed:         Photos → DepthFlow parallax clips → Agentado overlays → Video
```
Existing overlay pipeline (price intro, stats, end card, agent photo) stays intact. Only the animation engine changes.

### What This Won't Do
Trolto's "animate details" (flowing fountain, swaying curtains) requires generative image-to-video AI. Even cheap open models (Wan 2.1, CogVideoX) need beefy GPUs and cost ~$0.05-0.10/video on RunPod. Not viable for free tier at scale.

### What This WILL Do
Dramatic camera fly-throughs with real depth perception, seamless loops, artifact-free edges — a significant upgrade from flat Ken Burns at zero cost.

## 🎬 DepthFlow CPU Parallax Renderer — Built and Tested (02:00 UTC)

### DepthFlow on Hostinger — Failed
Tried installing DepthFlow directly on the Hostinger container. Failed because:
- No GPU or CUDA available
- No EGL support for offscreen GLSL rendering
- `/tmp` partition (10GB, 88% full) ran out of space during PyTorch+CUDA download (~1.5GB)
- Workaround: set `TMPDIR=/var/tmp` (500GB+ free), installed with `--break-system-packages`
- DepthFlow CLI installed but crashes with "No GPU Available" on both default and headless mode

### CPU-Based Parallax Renderer — Working
Built `/var/tmp/depthflow-test/parallax_render.py` — a self-contained CPU renderer using:
- **Depth estimation**: `depth-anything/Depth-Anything-V2-Small-hf` via HuggingFace transformers (~300MB model download, first run only)
- **Parallax effect**: OpenCV `cv2.remap()` with depth-boosted displacement + dolly zoom + camera sway
- **Encoding**: FFmpeg pipe (rawvideo → libx264)

### Two Iterations — Invisible → Visible

**v1 — Failed**: Displacement formula was `depth * intensity * 0.3` with intensity=0.03, yielding <1 pixel max displacement. Mean frame-to-frame diff was 2.1 (out of 255), max diff 167. Movement invisible to human eye.

**v2 — Working**: Rewrote with:
- Dolly zoom: 8% scale-up over clip duration
- Depth boosting: `np.power(depth, 1.5)` to exaggerate foreground/background separation
- Real pixel displacement: `depth * max_displacement * t` where max_displacement defaults to 100px
- Camera sway: sine-wave pan (15px horizontal, 10px vertical) for organic feel
- Smoothstep easing on all movement curves
- Streaming to FFmpeg (no in-memory frame buffer)

**Results**: 70% pixels changed >3 units, 46% >10 units, max shift 255px. 3 seconds @ 30fps rendered in ~37 seconds (plus ~20s depth estimation). Output: 1080×1920, ~400KB H.264.

### File Locations
- `/var/tmp/depthflow-test/parallax_render.py` — CPU renderer script
- `/var/tmp/depthflow-test/output/cpu_parallax.mp4` — output video
- `/tmp/agentado_depthflow.ipynb` — Colab notebook (deployed to `agentrocketman.com/agentado/depthflow-test.ipynb`)

## Session — 2026-07-14 21:30 UTC — Agentado Preview Generation: Root Cause + Fix

### The Real Root Cause
After multiple sessions of debugging (yuvj420p fix, serve.php proxy, tunnel rewiring), the preview STILL showed "Format error" in browser. The ACTUAL root cause was Cloudflare's `trycloudflare.com`:

1. **Cloudflare blocks MP4 GET downloads** — PHP did a POST to tunnel to generate video, then a separate GET to download the `.mp4`. Cloudflare intercepted the GET with browser-challenge HTML (not the actual video data). PHP saved that HTML as the `.mp4` → browser "Format error."

2. **Cloudflare also blocks POSTs on old tunnels** — The tunnel URL (`packing-alex-subtle-action`) had been flagged by Cloudflare. Even POSTs were getting intercepted. `GET /health` still worked (whitelisted pattern), but `POST /` returned "Not Found" (Cloudflare, not our Python server).

### The Fix — Base64 Video in JSON Response
Instead of two requests (POST to generate + GET to download), the Python server now:
1. Generates the video via FFmpeg
2. Reads the .mp4 file
3. Encodes it as base64
4. Returns it inline in the JSON response: `{ videoData: "<base64>", videoSize: N, jobId: "..." }`
5. PHP decodes the base64 and saves to Hostinger

**This eliminates Cloudflare entirely from the video download path.** Only the initial POST goes through the tunnel (which Cloudflare allows on fresh tunnels).

- Fresh cloudflared tunnel: `https://catalogs-associated-deadline-cornwall.trycloudflare.com`
