## What Mission Control Is
Multi-agent pipeline for autonomous code generation, phone-first workflow:
**Scout тЖТ Architect тЖТ Designer* тЖТ Tester тЖТ Reviewer тЖТ Builder тЖТ Code Reviewer тЖТ Smoke Test тЖТ Security* тЖТ Performance* тЖТ Deployed**

Stages with `*` are optional toggles. 8 planning agents are pure Sonnet API calls in PHP. Builder runs Claude Code CLI via container-side daemon.

## Key Architecture Decisions
- **Container-based Builder Runner** (`/data/.openclaw/workspace/builder-runner.py`) тАФ Python daemon polling Airtable every 15s for projects with `current_stage=builder`. Runs detached via setsid (PPID=1). Heartbeats to `/api/runner-heartbeat.php` with secret `mc-runner-heartbeat-2026`. Status pill in dashboard nav.
- **Single-file bulletproof HTML** тАФ no React/Vue/external deps in pages. Vanilla PHP backend.
- **Airtable as DB** тАФ base `apptYNRJTXwItvied`. Tables:
  - MissionControl_Projects: `tblZDjRO5OSIqzmEY`
  - MissionControl_AgentStatus: `tblwlhJRTnuHzivlb`
  - MissionControl_Approvals: `tblr4Wex6GwRwz4WE`
- **Stage tracking via `current_stage` text field** тАФ legacy `status` singleSelect couldn't be extended via API. New text field accepts any stage string.
- **Auto-approve toggles preset upfront** (Create/Detail modals), NOT per-card. Per-stage flags: `auto_scout`, `auto_code_auditor` (legacy name for code_reviewer), `auto_smoke_test`, `auto_security`, `auto_performance`.
- **Patch modes**: `full`, `quick_fix`, `bug_fix`, `add_feature` тАФ each has its own pipeline via `getPipelineStages()`.
- **Async agent firing**: CURLOPT_TIMEOUT_MS 300, fire-and-forget pattern.
- **Webhook auto-creates approvals** when status flips to `waiting_approval`.

**Project**: `recDZeVn8PrYtwMjF` (URL Shortener v1.1 (fix), patch_mode=quick_fix)
**Symptom**: Chris approved Scout recommendations but next stage didn't fire.

**Three bugs found & fixed**:
1. `patch.php` тАФ `getFirstStage()` got partial Airtable POST response missing `patch_mode`, defaulted to full mode тЖТ Scout instead of Builder. **Fix**: explicitly set `patch_mode` + enable flags on `$newProject['fields']` before calling.
2. `decide.php` тАФ when approved stage isn't in project's pipeline, `getNextStage()` returns null. **Fix**: fallback to `expectedStages[0]` from project's actual mode pipeline.
3. `run.php` тАФ `stage='builder'` returned "manual" message but never set `current_stage='builder'`, so Builder Runner never saw it. **Fix**: PATCH project to `current_stage=builder`, `status=builder`, `current_agent='Builder-Runner (queued)'` + post idle status event.

## Lessons Learned
- **Airtable POST returns partial fields** тАФ never trust the response shape; merge fields explicitly when chaining logic.
- **`getNextStage` needs fallback** тАФ pipelines vary per patch_mode; approving a stage outside the active pipeline must gracefully advance to next valid stage.
- **Async fire-and-forget needs visible side effect first** тАФ if you fire run.php for Builder and the only effect is queueing, you MUST flip `current_stage` first so the polling Runner sees it.
- **Phone-first means container-side runner** тАФ laptop daemons don't survive sleep/network changes. OpenClaw Docker container is always-on.
