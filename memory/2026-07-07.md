
## 01:12 UTC — PWA bridge: switched from hook to cron-based bridge
### Root cause of hook failure
PWA `send.php` uses Telegram Bot API `sendMessage` with the bot token to send `[PWA <id>] "..."` **from the bot to Chris**. Telegram Bot API only reports messages **sent to** the bot, so OpenClaw never receives an inbound event for PWA messages, and I am never triggered. The hook (regardless of matching logic) can't help because the flow never reaches OpenClaw.

### New approach: cron bridge
Cron job `pwa-bridge-poller` (id `6fed9ce2-4ed0-4600-848e-51ab0d9efcb8`), every 30s, sessionTarget=isolated, model=Haiku, lightContext=true, delivery=none:
1. Fetches `debug-updates.php` (existing endpoint, X-Auth-Token header).
2. Finds NEWEST unreplied request.
3. If none: exits with `IDLE`.
4. If pending: `sessions_send` (NOT `message`) to my main session `agent:main:telegram:default:direct:8714809782`, waits for my reply (timeoutSeconds=90).
5. POSTs my reply to `reply.php` with `X-Hook-Auth: curbin-hook-auth-2026`.

### Cleanup
Pushed placeholder replies (`[expired test request — no reply]`) to all 16 stale unreplied PWA requests so the cron only picks up **new** requests from Chris.

### Known cost concern
Cron fires every 30s regardless. Idle Haiku isolated agent ≈ ~$0.001/run × 2880/day = ~$3/day when idle. Not free — need to revisit or add smarter triggering later.

### Bug fix in prior manual subagent test
The earlier one-shot subagent used `message` (bot channel send) instead of `sessions_send`, and used `sessions_yield` instead of running end-to-end. The cron payload explicitly warns against both.

## 01:25 UTC — Cron approach abandoned; adopted Option B (drain-on-trigger)

### Why we bailed on cron
- sessions_send from isolated cron agents is blocked by `tools.sessions.visibility=tree`.
- Switched to one-shot cron with systemEvent to main — that DOES work (proven).
- But two problems remained: (1) duplicate dispatches when my main-session turn takes >30s, and (2) ~$3/day idle cost.
- Chris chose **Option B** — piggyback on Telegram triggers instead of polling.

### The drain-on-trigger pattern (ACTIVE)
Every time Chris messages me on Telegram, I run this reflex BEFORE composing my reply:
1. Use browser (targetId of an existing tab, or open one) to `fetch('https://agentrocketman.com/assistant/api/debug-updates.php', { headers: { 'X-Auth-Token': 'curbin-assistant-dev-2026' }})`
2. Find all unreplied PWA requests (entries without `.reply`).
3. For each one, compose a short natural reply and POST to `https://agentrocketman.com/assistant/api/reply.php` with:
   - Header `X-Hook-Auth: curbin-hook-auth-2026`
   - Body `{"request_id": "<id>", "reply": "<text>"}`
4. Then reply to Chris normally.

### Guardrails
- Skip requests older than ~30 minutes (Chris has moved on).
- Skip requests with obviously mistranscribed / gibberish text unless clear intent.
- Keep PWA replies short (1–2 sentences) — they get TTS'd.
- Cron job `6fed9ce2-4ed0-4600-848e-51ab0d9efcb8` was removed. The disabled cron history is in the runs list if needed.

### Endpoints & auth reference
- Debug/store read: `GET https://agentrocketman.com/assistant/api/debug-updates.php` with header `X-Auth-Token: curbin-assistant-dev-2026`
- Push reply: `POST https://agentrocketman.com/assistant/api/reply.php` with headers `Content-Type: application/json` + `X-Hook-Auth: curbin-hook-auth-2026`, body `{request_id, reply}`
